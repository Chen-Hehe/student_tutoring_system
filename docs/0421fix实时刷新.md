## 为什么我判断根因主要在后端推送，而不是前端渲染

你给的现象里有两个特别关键的点：

### 现象 1：接收方只能收到 `pong`
这说明：

- WebSocket 建连是成功的
- 心跳也是成功的
- 但业务消息没有通过同一连接发到前端

### 现象 2：接收方频繁请求 `/chat/conversations`，而接口响应里其实已经有新消息和未读数
你给的响应里已经有：

```json

  "userName": "张明华",
  "lastMessage": "buxing!",
  "unreadCount": 1
}
```

这说明：
- 数据库没问题
- 会话查询没问题
- 未读数计算没问题

只是页面没有通过 WebSocket 收到那条即时消息，所以 UI 没实时插入消息，只能靠刷新或重新请求来“间接看到”

所以问题不是“消息没存进去”，而是“**消息没从服务端推到客户端**”。

## 你原来后端实现里最可疑的点

原来的 `ChatWebSocketHandler.onRedisMessage()` 是这样处理的：

```103:118:backend/src/main/java/com/tutoring/handler/ChatWebSocketHandler.java
    public void onRedisMessage(byte[] messageBytes, String channel) {
        try {
            String message = new String(messageBytes);
            log.info("【Redis 收到消息】channel={}, message={}", channel, message);
            
            String channelId = channel.replace(REDIS_CHANNEL_PREFIX, "");
            Long userId = Long.parseLong(channelId);
            
            // 直接推送原始消息
            sendToUser(userId, message);
            log.info("【WebSocket 推送】已推送给用户 {}", userId);
```

这里的问题是：

- `messageBytes` 很可能不是你前端预期的纯净 JSON 文本
- `sendToUser(userId, message)` 又会再做一次 JSON 序列化
- 最终前端可能收到的是字符串包字符串，或者 Redis 序列化后的内容，不一定是 `JSON.parse` 后能得到标准消息对象

而你现在本地是单机开发，其实根本没必要先依赖 Redis 才把消息送给接收方。

所以我优先把**直接 WebSocket 推送**加上了，这是最稳的修法。
