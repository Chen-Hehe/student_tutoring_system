为了彻底解决**“重复发送”**和**“状态确认（真实 DB ID 缺失）”**的问题，同时为您后续加入图片、语音等富媒体消息打好基础，我强烈建议统一采用**“REST API（HTTP）发送，WebSocket 接收推送”**的架构链路。

以下是为您梳理的标准“消息发送链路”设计方案及具体的代码改造指南：

### 一、 核心架构定调：REST 发 + WS 收

为什么不推荐纯 WebSocket 发送？

因为您后续规划了**图片（type=2）**和**语音（type=3）**功能。文件上传通常需要通过 HTTP（如 OSS 服务）进行，如果使用 HTTP 上传文件再用 WebSocket 发送消息，逻辑会割裂。统一走 HTTP POST 接口发送消息，可以极其方便地配合 Multipart 表单上传，代码最简洁、最稳定。

---

### 二、 标准消息发送链路（时序流程）

整个流程分为 **前端乐观更新 -> 后端落库推送 -> 前端状态确认** 三步：

1. **用户操作**：用户点击“发送”。
    
2. **生成临时消息（乐观更新）**：前端生成一个临时 ID（如 `temp_1700000000`），将消息立即展示在聊天窗口，状态设为 `[发送中...]`。
    
3. **HTTP 接口请求**：前端调用 `POST /api/chat/send`。
    
4. **后端落库**：后端 `ChatController` 接收请求，将消息存入 `chat_records` 表，数据库生成真正的**主键 ID** 和 **创建时间（timestamp）**。
    
5. **后端触发推送**：后端将带有真实 ID 的消息体发送到 Redis 通道（`chat:user:{receiverId}`）。
    
6. **响应前端（ACK 确认）**：HTTP 接口直接返回 `200 OK`，并将完整的消息对象（带真实 DB ID）返回给发送方。
    
7. **前端替换 ID**：发送方前端拿到 API 响应后，在本地列表中找到刚才那个 `temp_ID` 的消息，将其替换为真实的 ID，状态改为 `[发送成功]`。
    
8. **接收方接收**：接收方所在节点的 WebSocket 监听到 Redis 消息，将消息 Push 给接收方的浏览器，接收方界面更新。

#### 2. 教师端改造 (解决“状态不确认 / tempId 残留”问题)

教师端目前采用的是纯 WebSocket 发送且缺少确认。请改为和学生端一致的 HTTP 发送模式，并完善乐观更新逻辑。

**修改 `teacher-frontend/src/pages/Chat.jsx` 中的 `sendMessage` 方法：**

JavaScript

```
const sendMessage = async () => {
  // ... 前置校验保持不变 ...
  setSending(true)
  
  // 1. 生成临时 ID，进行乐观更新（让 UI 瞬间响应）
  const tempMessageId = 'temp_' + Date.now()
  const newMessage = {
    messageId: tempMessageId, // 临时 ID
    senderId: currentUser.id,
    receiverId: selectedConversation.userId,
    message: inputValue.trim(),
    type: 1,
    status: 'sending' // 可在 UI 渲染时加上 loading 小圆圈
  }
  
  setMessages(prev => [...prev, newMessage]) // 立即上屏
  setInputValue('') // 清空输入框
  
  try {
    // 2. 统一改为 HTTP 调用！
    // ❌ 删除 wsService.send(messageData)
    const result = await chatAPI.sendMessage({
      receiverId: selectedConversation.userId,
      message: newMessage.message,
      type: 1
    })
    
    // 3. API 返回成功，将刚上屏的临时消息替换为包含真实 DB ID 的消息
    const realMessage = result.data;
    setMessages(prev => prev.map(msg => 
      msg.messageId === tempMessageId 
        ? { ...realMessage, status: 'success' } // 替换真实 ID 和状态
        : msg
    ))
    
    loadConversations()
  } catch (error) {
    // API 发送失败，将该消息状态改为 error，允许用户重试
    setMessages(prev => prev.map(msg => 
      msg.messageId === tempMessageId 
        ? { ...msg, status: 'error' } 
        : msg
    ))
    antdMessage.error('发送失败，请重试')
  } finally {
    setSending(false)
  }
}
```

#### 3. 后端架构抽离 (统一分发入口)

因为消息改为从 HTTP Controller 进入，所以我们需要把原本写在 `ChatWebSocketHandler` 里的发布逻辑抽离出来，让 Controller 也能调用。

**推荐改造步骤：**

1. **清理 WebSocket 接收端**：修改 `ChatWebSocketHandler.java` 中的 `handleTextMessage` 方法。如果前端不再通过 WS 发送聊天文本，您可以将此方法精简为只处理心跳 (`ping/pong`)，**不再处理消息入库**。
    
2. **提取发布逻辑**：在 `ChatRecordService` (或独立的 Message 调度服务) 中实现真正的业务：
    
    Java
    
    ```
    @Transactional
    public ChatRecord saveAndPushMessage(ChatMessage chatMessage) {
        // 1. 保存到 MySQL 数据库
        ChatRecord record = new ChatRecord();
        // ... 实体转换赋值 ...
        chatRecordRepository.save(record);
    
        // 回填真实 DB ID 和生成的时间
        chatMessage.setId(record.getId());
        chatMessage.setTimestamp(record.getSentAt());
    
        // 2. 发送到 Redis 频道，让 WebSocket 监听器去广播
        String channel = "chat:user:" + chatMessage.getReceiverId();
        redisTemplate.convertAndSend(channel, objectMapper.writeValueAsString(chatMessage));
    
        return record;
    }
    ```
    
3. **Controller 调用**：`ChatController.java` 的 `/send` 接口直接调用 `saveAndPushMessage`，并把落库后的 `record` 返回给前端即可。
    

### 四、 总结与好处

采用上述链路重构后，系统将获得以下收益：

1. **数据绝不重复**：所有写入操作只走 HTTP 一条线，彻底杜绝 WebSocket 导致的并发双重写入。
    
2. **前端状态 100% 同步**：前端有了确切的 `API Response`，能够拿到真实的数据库 ID。以后做**消息撤回、消息已读回执**时，前端才能精确传给后端 `messageId`。
    
3. **完美兼容后续多媒体**：将来做发图片功能时，只需：`前端调 OSS 传图 -> 拿到 URL -> 调用 sendMessage API 传入 fileUrl -> 后端落库 -> WS推送到对方`，逻辑无比丝滑。
